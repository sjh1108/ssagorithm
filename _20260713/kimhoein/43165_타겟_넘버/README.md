# 타겟 넘버

- 문제: [프로그래머스 43165번 - 타겟 넘버](https://school.programmers.co.kr/learn/courses/30/lessons/43165)
- 유형: DFS, 완전 탐색
- 코드: [타겟_넘버.java](./타겟_넘버.java)

## 내가 이해한 문제

- `numbers` 배열의 각 숫자 앞에 `+` 또는 `-`를 붙일 수 있다.
- 모든 숫자를 사용해서 `target`을 만드는 경우의 수를 구한다.
- 모든 `+`, `-` 조합을 확인해야 하므로 완전 탐색 문제다.
- 모든 경우를 DFS로 탐색할 수 있다.

## 내가 처음 잡은 풀이 방향

- 각 숫자마다 더하는 경우와 빼는 경우를 모두 탐색한다.
- 배열의 끝까지 갔을 때 현재 합이 `target`과 같으면 `count`를 증가시킨다.
- `depth == numbers.length`가 되면 탐색을 종료한다.

## 모델링 정리

### 상태

- `depth`: 현재 몇 번째 숫자를 보고 있는지
- `sum`: 지금까지 만든 합

### 전이

현재 숫자를 더하거나 뺀 뒤 다음 숫자로 넘어간다.

```text
(depth, sum) -> (depth + 1, sum + numbers[depth])
(depth, sum) -> (depth + 1, sum - numbers[depth])
```

### 제약

- `depth == numbers.length`가 되면 더 이상 탐색하지 않는다.
- 각 숫자는 주어진 순서대로 한 번씩 사용한다.

### 목표

- 모든 숫자를 사용했을 때 `sum == target`이면 경우의 수를 1 증가시킨다.
- 모든 탐색이 끝나면 가능한 경우의 수를 반환한다.

## 헷갈렸던 부분과 피드백

- 처음에는 브루트포스로만 보였지만, 모든 경우를 확인해야 한다는 판단은 맞았다.
- 브루트포스는 모든 경우를 확인하는 전략이고, DFS는 그 경우를 탐색하는 구현 방식이다.
- `count`는 상태가 아니라 정답을 누적하는 변수다.
- 이 문제의 상태는 현재 몇 번째 숫자인지를 나타내는 `depth`와 현재까지 계산한 값인 `num`이다.
- 코드의 `num`은 현재 숫자가 아니라 누적 합을 의미하므로 `sum`이라는 이름을 사용하면 역할이 더 잘 드러난다.
- 정적 누적 변수인 `count`는 `solution()`이 여러 번 호출되는 환경이라면 탐색 전에 `0`으로 초기화할 필요가 있다.

## 제출 코드에 대해

- 위 피드백은 별도로 정리하되, Java 파일에는 문제를 풀었을 때 직접 작성한 코드를 그대로 남겼다.
- 코드의 동작은 바꾸지 않고 들여쓰기와 공백만 정리했다.

## 이 문제에서 얻은 포인트

- 완전 탐색과 DFS는 서로 다른 개념이다.
- 완전 탐색은 모든 경우를 확인하는 전략이다.
- DFS는 한 경로를 끝까지 탐색한 뒤 돌아오는 방식이다.
- 이 문제의 상태는 `sum`만으로 부족하고 현재 몇 번째 숫자인지도 필요하다.
- 따라서 상태는 `depth`와 `sum`으로 표현할 수 있다.
- 재귀 함수의 매개변수를 살펴보면 탐색 상태를 파악하기 쉽다.
