# 주문 취소가 있는 매칭 엔진 — 풀이 (의도와 목적)

## 문제 핵심
A1의 매칭 엔진에 **주문 취소(`CANCEL id`)**가 추가됩니다. 취소된 주문은 이후 체결에 참여하지 않고, 최종 대기 목록에서도 빠집니다. 명령이 `LIMIT`(신규)과 `CANCEL`(취소) 두 종류로 나뉩니다.

## 어떻게 접근했는가 — "힙에서 임의 원소를 지운다"는 난관
A1과 달리, 취소는 **힙 한가운데 있는 임의의 주문을 없애는 일**입니다. 그런데 힙은 top만 O(log n)에 다루도록 설계된 자료구조라, 내부 임의 원소를 직접 찾아 삭제하는 연산이 없습니다(있어도 위치 추적에 비용이 큽니다).

여기서 택한 전략이 **지연 삭제(lazy deletion)**입니다. 아이디어는 간단합니다.

> 취소 시점에는 힙을 건드리지 않는다. 대신 그 주문에 "취소됨" 표시만 남긴다. 나중에 그 주문이 힙의 **top으로 올라왔을 때** 비로소 버린다.

취소는 O(1)에 표시만 하고, 실제 제거 비용은 어차피 한 번은 top을 지나가며 pop되는 흐름에 자연스럽게 흡수됩니다.

이를 위해 **`id → 주문 객체` 해시맵**이 필요합니다. `CANCEL id`가 오면 맵에서 주문을 O(1)로 찾아 플래그를 세웁니다.

## 왜 이렇게 설계·구현했는가
`OrderBook`에 다음을 더했습니다.
- `byId` : 취소 대상을 O(1)로 찾기 위한 `HashMap<id, Order>`
- `canceled` 플래그 + 취소 시 `qty = 0` : 체결 로직이 "잔량 0"만 봐도 자연스레 건너뛰도록
- `cleanAsk/cleanBid` : top이 **수량 0이거나 취소됨**이면 계속 pop하여, 항상 "살아 있는" 최우선 호가만 보이게 함

```java
void cancel(int oid) {
    Order o = byId.get(oid);
    if (o != null && !o.canceled && o.qty > 0) { o.canceled = true; o.qty = 0; }
}
void cleanAsk() {
    while (!asks.isEmpty() && (asks.peek().qty == 0 || asks.peek().canceled)) asks.poll();
}
```

체결 루프는 매번 먼저 `cleanAsk()`(또는 `cleanBid()`)로 죽은 top을 정리한 뒤 최우선 호가를 봅니다. 덕분에 "취소된 주문과는 절대 체결되지 않는다"가 코드 한 줄로 보장됩니다.

## 놓치기 쉬운 포인트
- **id와 시간 우선순위(도착순서)는 별개**입니다. id는 취소 지정용 식별자일 뿐이고, 힙의 tie-break는 반드시 **도착순서(seq)**로 해야 합니다. (그래서 `seq`를 따로 둡니다.)
- 최종 대기 목록을 뽑을 때도 `qty > 0 && !canceled`로 걸러야 지연 삭제로 남아 있던 유령 주문이 새지 않습니다.
- 취소는 O(1) 표시, 정리는 top에서 한 번 → 전체적으로 추가 비용이 상수에 가깝습니다.

## 복잡도
지연 삭제로 각 주문은 힙에 한 번 push, 한 번 pop 됩니다. 취소·조회는 해시맵 O(1). 전체 **O(N log N)**.
