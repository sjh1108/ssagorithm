# 자동 전투 시뮬레이터 — 풀이 (의도와 목적)

## 문제 핵심
두 팀의 유닛이 라운드마다 **속도 순으로** 한 번씩 자동 행동합니다. 병종(전사·궁수·성직자)에 따라 행동이 다릅니다. 한 팀이 전멸하면 즉시 종료, R라운드까지 안 끝나면 무승부입니다. 최종 결과·경과 라운드·각 유닛 잔여 체력을 출력합니다.

## 어떻게 접근했는가 — 분기 대신 다형성
가장 위험한 선택은 행동 처리를 `if (전사) … else if (궁수) … else if (성직자) …`로 늘어놓는 것입니다. 병종이 하나 늘 때마다 이 분기가 곳곳에 번지고, 후속 문제(상태이상·격자)에서 그대로 무너집니다.

핵심 관찰은 **"모든 유닛은 자기 차례에 `행동한다`는 공통 계약을 가지되, 그 내용만 병종마다 다르다"**는 점입니다. 이건 다형성의 교과서적 상황입니다.

> `Unit`을 추상 타입으로 두고 `act(Battle)`만 선언한다. 전사/궁수/성직자는 각자 `act`를 **오버라이드**한다. 전투 루프는 `u.act()`만 호출할 뿐, 그 유닛이 무엇인지 모른다.

이렇게 하면 새 병종은 클래스 하나만 추가하면 되고, 전투 진행 코드는 손대지 않습니다(개방-폐쇄).

## 왜 이렇게 설계·구현했는가
- `Unit` (추상): 체력·공격력·방어력·속도 + `abstract act(Battle)`.
- `Warrior/Archer/Priest`: 대상 선정 규칙만 다르게 오버라이드.
  - 전사 = 적 최저 HP 공격(마무리 담당), `max(1, 공격력−방어력)`
  - 궁수 = 적 최고 공격력 저격, **방어 무시**(dmg = 공격력)
  - 성직자 = 아군 중 **체력 비율 최저** 치유, 전원 만피면 약공격
- `Battle`: `enemyMinHp`, `enemyMaxAtk`, `allyMinRatio` 같은 **선택 헬퍼**를 모아 둡니다. 각 병종은 "누구를 고를지"만 헬퍼에 위임하고 "무엇을 할지"에 집중합니다.

시뮬레이션 루프의 뼈대:

```java
for (int rnd = 1; rnd <= rounds; rnd++) {
    Unit[] order = liveOrder();                 // (-속도, 번호)로 정렬
    for (Unit u : order) {
        if (u.hp <= 0) continue;                // 이번 라운드 먼저 죽었으면 스킵
        u.act(this);                            // 다형적 호출
        if (!teamAlive(0) || !teamAlive(1))     // 행동마다 전멸 확인
            return new int[]{ teamAlive(0) ? 0 : 1, rnd };
    }
}
return new int[]{ 2, rounds };                  // 무승부
```

## 놓치기 쉬운 포인트 — 결정론
시뮬레이션 문제는 **모든 동점·순서가 유일하게 정해져야** 정답이 하나로 확정됩니다.
- 행동 순서 동점: 속도가 같으면 **먼저 입력된(번호 작은)** 유닛 우선.
- 대상 동점: "가장 ~한" 대상이 여럿이면 **번호 최소**.
- **체력 비율 비교는 정수 교차곱**으로: `a.hp * b.max` vs `b.hp * a.max`. 부동소수점(`hp/max`)을 쓰면 동점 판정이 미세하게 어긋나 오답이 납니다.
- 전멸 확인은 **매 행동 직후**. "라운드 끝에 한 번"이 아니라, 마지막 적을 쓰러뜨린 그 순간 종료해야 경과 라운드가 맞습니다.

## 복잡도
라운드 R, 유닛 n에 대해 각 행동이 상대 팀을 한 번 훑습니다. **O(R · n²)** — 자바에서는 최악(예: 100v100, R=1000)도 0.3초 안쪽입니다.
