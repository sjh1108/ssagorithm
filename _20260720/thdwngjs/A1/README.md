# 지정가 주문 매칭 엔진 — 풀이 (의도와 목적)

## 문제 핵심
주문이 하나씩 도착합니다. 매수는 "이 가격 **이하**면 산다", 매도는 "이 가격 **이상**이면 판다"입니다. 새 주문이 오면 반대편 호가창에서 가장 유리한(매수엔 최저 매도가, 매도엔 최고 매수가) 주문과, 같으면 **먼저 온** 주문과 조건을 만족하는 한 계속 체결하고, 남은 수량은 호가창에 등록합니다. 마지막에 체결 내역·대기 잔량·총 체결금액을 출력합니다.

## 어떻게 접근했는가
이 문제의 병목은 매 주문마다 **"최우선 호가 하나를 찾는 일"**입니다. 대기 주문을 리스트에 담고 매번 최적가를 선형 탐색하면 주문당 O(n), 전체 **O(n²)** 이라 N=10만에서 무너집니다.

핵심 관찰은 **매번 필요한 건 "가장 좋은 호가 하나"뿐**이라는 점입니다. 전체를 정렬된 상태로 유지할 필요가 없습니다. 그래서 최우선 원소만 O(log n)에 꺼내는 **힙(우선순위 큐)**이 정확히 맞습니다.

- 매수 대기: **최고가**가 top인 최대 힙
- 매도 대기: **최저가**가 top인 최소 힙
- 동가는 먼저 온 주문 우선이므로, 우선순위 키를 `(가격, 도착순서)`로 잡습니다.

## 왜 이렇게 설계·구현했는가
자료구조 두 개(매수 힙·매도 힙)를 `OrderBook`으로 **캡슐화**하고, 바깥에는 "주문을 받아 체결시키고 잔여를 등록"하는 `add` 하나만 노출했습니다. 호출부는 호가창 내부가 힙 두 개라는 사실을 몰라도 됩니다. 주문 하나는 `Order`(도착순서·방향·가격·잔량) 객체이고, **도착순서(idx)가 곧 시간 우선순위**입니다.

체결 루프는 규칙을 그대로 옮긴 형태입니다(매수 기준):

```java
while (o.qty > 0) {
    Order best = bestAsk();                          // 최저가 매도
    if (best == null || best.price > o.price) break; // 조건 미달이면 종료
    int t = Math.min(o.qty, best.qty);               // 체결량 = 더 적은 쪽
    trades.add(new int[]{best.price, t});            // 체결가 = '먼저 온' 대기 주문 가격
    o.qty -= t; best.qty -= t;
    if (best.qty == 0) asks.poll();                  // 소진되면 호가창에서 제거
}
if (o.qty > 0) bids.add(o);                          // 남으면 대기로 등록
```

`bestAsk()`는 top에 남은 "수량 0" 주문을 걷어낸 뒤 최적가를 돌려줍니다(이 문제엔 취소가 없어 0만 정리).

## 놓치기 쉬운 포인트
- **체결 가격은 신규 주문 가격이 아니라, 먼저 대기하던 주문의 가격**입니다(가격-시간 우선 원칙).
- **동가 시간 우선**: 가격이 같으면 도착순서가 빠른 주문이 top이어야 합니다. Comparator의 2차 키로 반드시 넣습니다.
- **총 체결금액 오버플로**: 가격·수량이 각각 최대 100만이라 합이 32비트를 넘습니다. `long`으로 누적합니다.

## 복잡도
각 주문은 힙에 최대 한 번 들어가고 한 번 빠집니다. 전체 **O(N log N)**.
