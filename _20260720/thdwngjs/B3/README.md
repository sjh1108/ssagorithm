# 격자 위의 자동 전투 — 풀이 (의도와 목적)

## 문제 핵심
전투가 $H \times W$ 격자 위에서 벌어집니다. 유닛은 위치를 가지고, 매 턴 **이동 / 공격 / 치유 중 하나**를 합니다. 병종마다 사거리가 다르고(전사 1, 궁수 3, 성직자 2, 맨해튼), 사거리 안에 대상이 없으면 장애물을 피해 **최단 경로로 접근**합니다.

## 어떻게 접근했는가 — 두 개의 "거리"를 분리하라
이 문제의 함정은 "거리"가 두 종류라는 점입니다.
- **위협 판단(공격·치유 사거리)**: 화살과 치유는 벽을 넘습니다 → **맨해튼 거리**로 판단.
- **이동**: 벽과 다른 유닛을 통과할 수 없습니다 → **격자 최단 경로(BFS)**로 판단.

이 둘을 섞으면(예: 이동 거리로 사거리를 재거나, 맨해튼으로 이동시키면) 곧바로 틀립니다. 그래서 **"쏘는 거리는 맨해튼, 걷는 거리는 BFS"**로 명확히 분리하는 것이 설계의 출발점입니다.

두 번째 관찰: 근접·원거리 유닛이 대상을 못 찾았을 때의 "접근"은 공통 동작입니다.

> 접근이란 "적에게 인접한 빈칸(=공격 위치) 집합으로 가는 최단 경로에서 한 칸 전진"이다. 이 목표 칸 집합에서 **다중 소스 BFS**를 한 번 돌리면 격자 전체의 "목표까지 거리장"이 나오고, 내가 선 칸의 거리 $D$에서 $D-1$인 인접 칸으로 가면 된다.

## 왜 이렇게 설계·구현했는가
B1의 전투 코어(다형성)를 그대로 두고, `Unit`에 좌표를 더하고 `Battle`에 격자·점유·BFS를 얹었습니다. 각 병종 `act()`는 "사거리 안에 대상 있으면 공격/치유, 없으면 `approach()`" 구조입니다.

`approach()`가 이 문제의 심장입니다.

```java
void approach(Unit mover, Set<Integer> targets) {   // targets = 공격 위치(빈칸)들
    // 목표 칸들을 소스로 다중 소스 BFS → 거리장 dist
    for (int key : targets) { dist[r][c] = 0; dq.add(...); }
    while (!dq.isEmpty()) { /* 상하좌우로 확장, 벽·타 유닛은 통과 불가 */ }

    int D = dist[mover.r][mover.c];
    if (D == INF || D == 0) return;                 // 도달 불가면 정지
    for (int d = 0; d < 4; d++) {                   // 상·하·좌·우 순서로
        int nr = ..., nc = ...;
        if (passable(nr,nc,mover) && dist[nr][nc] == D - 1) { /* 한 칸 이동 */ return; }
    }
}
```

## 놓치기 쉬운 포인트 — 결정론의 핵심들
- **방향 우선순위 고정**: 최단 경로가 여럿일 때 어디로 갈지 정하는 규칙이 없으면 답이 갈립니다. **상·하·좌·우** 순서로 $D-1$ 칸을 골라 유일하게 만듭니다.
- **비율 비교는 정수 교차곱**(성직자 치유 대상), 대상 동점은 **번호 최소** — B1·B2와 동일.
- **이동은 순차 반영**: 한 유닛이 움직이면 점유가 즉시 갱신되어, 같은 라운드 뒤 유닛의 BFS에 반영됩니다. (동시 이동으로 처리하면 충돌·오답)
- **죽은 유닛의 칸은 라운드 끝까지 점유로 남깁니다.** 한 유닛이 죽어도 그 시체가 그 라운드 동안 길을 막습니다. 이 타이밍을 어기면 미세하게 경로가 달라집니다.
- 공격 위치(목표 칸)는 "적에 인접하고 비어 있는 칸"이어야 합니다. 벽이거나 누가 서 있으면 목표가 될 수 없습니다.

## 복잡도
유닛마다 턴당 BFS 한 번(격자 $HW$). 라운드 R, 유닛 n에 대해 **O(R · n · HW)**. 제약($H,W\le20$, $n\le40$, $R\le500$)에서 자바로는 0.1초 수준입니다.
